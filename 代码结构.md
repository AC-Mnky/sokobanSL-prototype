# 代码结构（整理稿）

## 0. 目标
- 用最小模块集实现 NOSL 推箱子规则、模拟、求解。
- 规则层与 UI 层解耦；求解器复用同一规则层。

## 1. 分层与依赖
- `model`: 纯数据定义/复制/比较/序列化；无规则。
- `core`: 移动、按钮下降沿、事件轮次、写冲突裁决。
- `game`: 单步输入驱动（action -> next_state）。
- `solver`: BFS 生成器；调用 `core` 展开状态图。
- `io`: 关卡读写、静态元信息构建。
- `ui`: pygame 渲染与输入采集。
- 依赖方向：`ui -> game -> core -> model`，`solver -> core`，`io -> model`。

## 2. 数据模型
- `Coord = tuple[int, int]`（可负）。
- `Action = tuple[int, int]`，限定 `{(1,0),(0,1),(-1,0),(0,-1)}`。
- `Color = int`，`0` 为普通箱子色。
- `State = dict[Coord, MonoData | None]`，value 为 `None` 表示该坐标无数据。
- 约定：`value is None` 视为 `is_empty == True`。
- 对真实世界 `State`（关卡主状态）：
  - 键为**世界坐标**。
  - 移动、推入、写过程中，访问到 `coord not in state` 时，在该坐标生成 `value=None`。
  - `coord not in state` 与“已存在且 `value=None`”在机制语义上不声明等价（未来可区分）。
- 磁盘 `MonoData.data`（嵌套 `State`）：
  - 键为**相对磁盘所在世界格的偏移** `(rx, ry)`，对应世界格 `(Dx+rx, Dy+ry)`（磁盘在 `(Dx, Dy)`）。
- `InitialState: State`。
- 一个关卡定义为 `(static_state: StaticState, initial_state: InitialState)`。
- 磁盘初始 `data` 通常为“key 非空、value 全为 `None`”的字典（键为相对偏移）。
- `StaticState`:
  - `targets: dict[Coord, TargetData]`
  - `buttons: dict[Coord, list[ButtonData]]`
- `TargetData`:
  - `required_is_controllable: bool`
  - `required_color: int`
- `ButtonData`:
  - `button_type`（`s` 或 `l`）
  - `color: int`
  - 约束：禁止 `color == 0`
- `Event`:
  - 当前版本：`Event = ButtonData`（按钮事件）。
  - 扩展约束：后续新增事件类型时，`Event` 作为统一事件抽象保留不变。
- `MonoData`:
  - `is_empty: bool`（空气标记；true 时其余字段无效）
  - `is_wall: bool`（不可推动）
  - `is_controllable: bool`（角色）
  - `color: int`
  - `reject_save: bool`（默认 false；实体为真时 S 读阶段跳过该世界格，保留磁盘旧记录）
  - `reject_load: bool`（默认 false；实体为真时 L 写阶段跳过该世界格）
  - `data: State | None`（磁盘记录；`None` 等价空字典）
- 不变量：
  - `is_empty == True` 不可与 `is_wall/is_controllable` 同时生效。
  - 同一坐标最终至多 1 个 `MonoData`。
  - 不可推动与墙完全等价：`is_wall == True` 是唯一不可推动条件。

## 3. 关卡静态元信息（不参与 SL 覆盖）
- `StaticState.targets` 保存各种目标位置与目标条件。
- `StaticState.buttons` 保存同坐标可重叠的 `s/l` 按钮列表。
- 同一坐标的按钮列表中，每个按钮在一次下降沿都各触发一次（包括同色重复项）。

## 4. 规则流水线（单次玩家输入）
- 阶段 A：同步移动所有 `is_controllable` 对象。
  - 支持一条线上推动任意数量箱子。
  - 若目标链被墙阻塞，则该推动失败（越界不阻塞）。
  - 越界访问会触发对 `coord not in state` 的访问并生成 `value=None`，从而该格视为空气，可被推入。
  - 由此可知，玩家在地图边界外移动时，会在路径上留下由 `value=None` 形成的空气轨迹。
  - 某个角色被堵住时，仅该角色不动，其他角色照常移动。
  - 角色总是同向且同时移动，不会出现两个角色进入同一空格。
  - 同线平行移动时，即使角色紧邻或中间被箱子填满，只要前面的可移动，后面的也可移动（同时移动，不追尾）。
- 阶段 B：事件循环（下降沿驱动）。
  - `pending_events = collect_edge_events(prev_state, curr_state, static_state)`。
  - 下降沿判定仅依赖 `is_empty`：同一按钮格满足“前态 `is_empty=True`，后态 `is_empty=False`”即触发。
  - 当 `pending_events` 非空时，执行一轮“先读后写”：
    - 读阶段：对全部 `S/L` 事件读取源数据，生成候选写集合 `writes`。
    - 写阶段：调用统一写接口一次性提交 `writes`（内部完成冲突裁决）。
  - 提交后再次检测下降沿，得到下一轮 `pending_events`。
  - 循环直到 `pending_events` 为空（或进入无限连锁）。
- 注：无限连锁视为允许行为，不在规则层做修复。

## 5. S/L 语义
- `S(color)`：
  - 读：对每个相对键 `rel`，读世界格 `disk_pos + rel` 的当前 `MonoData`（`disk_pos` 为该磁盘在主 `State` 中的坐标）。
  - 快照仍写入磁盘 `data`，键保持为相对键 `rel`（与模板 key 集一致）。
  - 若该世界格在 `state.get(...)` 上为“无格/空”（`None`），保留磁盘该 `rel` 的旧记录。
  - 若该世界格为实体且 `reject_save`，保留磁盘该 `rel` 的旧记录（不读世界）。
  - 写：写到对应颜色磁盘所在格的 `data`。
- `L(color)`：
  - 读：读取对应颜色磁盘 `data`（相对键）。
  - 写：将读到的数据解包覆盖世界区域：对每个 `rel -> value`，写到世界格 `disk_pos + rel`；排除 value 为 `None` 的坐标；若目标世界格为实体且 `reject_load`，跳过该坐标。
  - 若磁盘无记录（`data is None` 或等价空），不生效。
- 无数据语义：
  - 若记录态某格值为 `None`，表示“不写该格”（跳过写操作）。
  - 空气是有效写入值（`is_empty=True`），与 `None` 区分。
- 同色多触发规则：
  - 每个被按下的按钮，会触发每个同色磁盘各一次。
  - 例：2 个红色磁盘 + 2 个红色按钮同轮下降沿，共触发 4 次 S/L。

## 6. 写冲突裁决
- 该逻辑是统一写提交接口的内部步骤，不单独暴露公共接口。
- 输入：同轮候选写 `writes: list[State]` 与提交前旧状态。
- 对每个 `coord` 聚合候选值并计数：
  - 值相等判定使用深比较（递归比较 `MonoData` 与其 `data`）。
  - 唯一最高频值：提交该值。
  - 最高频并列：该 `coord` 放弃写入，保留旧值。

## 7. 状态等价与哈希
- BFS 判重仅使用“可变信息”：
  - 各坐标 `MonoData` 的有效字段值。
  - 磁盘 `data` 的递归内容。
- 不纳入判重：
  - 关卡静态元信息（目标点、按钮布局、颜色映射）。
- 序列化建议：
  - 将 `State` 规约为按 `Coord` 排序的不可变元组。
  - `data` 递归同构序列化；`None` 与空气保留区分标记。

## 8 目标判定约束
- 对每个 `TargetData` 对应坐标：
  - 若该格 `is_empty == True`，则判定失败。
  - `required_is_controllable` 与 `required_color` 必须同时匹配当前格。

## 9. 最小接口
- `apply_action(state: State, action: Action, static_state: StaticState) -> State`
- `collect_edge_events(prev_state: State, next_state: State, static_state: StaticState) -> list[Event]`
- `build_event_writes(state: State, events: list[Event], static_state: StaticState) -> list[State]`
- `commit_writes(state: State, writes: list[State]) -> State`
- `run_event_cycle(prev_state: State, curr_state: State, static_state: StaticState) -> State`
- `is_goal(state: State, static_state: StaticState) -> bool`

## 10. 求解器接口契约
- `solve(initial_state: State, static_state: StaticState, step_chunk: int = 1024) -> Generator[tuple[str, int, tuple[Action, ...], int], None, None]`
- 每遍历 `step_chunk` 个状态至少 `yield` 一次：
  - `("solving", steps, (), searched_state_count)`：未找到解；且已知不存在 `<= steps` 步解。
  - `("solved", steps, solution, searched_state_count)`：找到解；`solution` 为动作序列。
- `searched_state_count` 始终为“已出队/已遍历状态数”。
- 无限连锁可能导致不可停机；该行为按规则保留，不额外定义 BFS 规避策略。
